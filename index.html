Twice a day, a frozen model publishes the probability that each Dublin Bikes station will have a bike, and a free dock, at the next commute window. Every forecast is committed to a public git ledger before its target time exists, then scored against what actually happened. The track record cannot be edited, selected, or quietly restarted; the git history would show it.

Pre-registered gate: after 28 scored days, the model must beat both baselines — station climatology and persistence — at the overnight horizon, with day-clustered bootstrap 95% confidence intervals excluding zero, for bikes and for docks. Anything less is published here as NOT PROVEN, in this same box. The gate was committed before any data was collected.

§2 How it works

01POLL

A cron job records every station's state each 10 minutes from the public GBFS feed.

02FORECAST

At 07:00 and 16:00 Dublin time, the frozen model turns current state, trend, calendar and weather into probabilities.

03COMMIT

Forecasts are appended to the ledger and pushed. The git timestamp is the before-the-fact proof.

04SCORE

Nightly, matured forecasts meet reality. Gaps and excluded stations are counted, never hidden.

§5 Verify this yourself

  1. Open any file in ledger/forecasts and note a forecast's target time.
  2. Find the commit that introduced it in the commit history: it predates the target.
  3. Check the outcome in ledger/outcomes and the running totals in summary.json — this page renders those same files, nothing else.
  4. Inspect the frozen model manifest and the offline validation report it had to pass before going live.
